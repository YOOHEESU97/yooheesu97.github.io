Cursor 쓰다 보면 모델이 너무 많아서 "이 상황엔 뭐 쓰지?" 하게 됩니다. 저도 처음엔 그냥 기본값만 썼는데, 쓰다 보니 작업마다 체감 차이가 꽤 크더라고요. 지금은 아래 표를 기준으로 골라 쓰고 있습니다.

| 상황 | 모델 |
| --- | --- |
| 평소 코딩 질문 | Composer 2.5 |
| 설명을 자세히 듣고 싶을 때 | Sonnet 4.6 High |
| 빠르게만 처리할 때 | Codex 5.3 High Fast |
| 어려운 설계 / 디버깅 | Opus 또는 GPT-5.5 |

## 사용 팁

- **일반 구현**은 Composer 2.5로 충분한 경우가 많습니다. 가볍게 물어볼 때 제일 무난해요.
- **아키텍처 결정**이나 **원인 불명 버그**는 reasoning 모델이 유리합니다. "왜?"를 파고들 때 차이가 납니다.
- 속도가 중요한 반복 작업은 Fast 계열을 씁니다. 린트 고치기, 보일러플레이트 생성 같은 거요.

모델 이름은 Cursor 기준이며, 다른 도구에서는 이름이 다를 수 있습니다. 핵심은 **"속도 vs 깊이"** 트레이드오프를 의식하는 것 같아요.

## 한 줄 정리

> 매번 최고 사양 모델 쓸 필요는 없습니다. 평소엔 가볍게, 막히면 무겁게 — 이 패턴이 제일 효율적이더라고요.
