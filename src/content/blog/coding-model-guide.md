Cursor 쓰다 보면 모델이 너무 많아서 "이 상황엔 뭐 쓰지?" 하게 됩니다. 저도 처음엔 그냥 기본값만 썼는데, 쓰다 보니 작업마다 체감 차이가 꽤 크더라고요. 지금은 아래 표를 기준으로 골라 쓰고 있습니다.

| 상황 | 모델 | 이유 |
| --- | --- | --- |
| 평소 코딩 질문 | Composer 2.5 | 속도·품질 균형, 일상 구현에 무난 |
| 설명을 자세히 듣고 싶을 때 | Sonnet 4.6 High | 추론·설명 품질이 좋음 |
| 빠르게만 처리할 때 | Codex 5.3 High Fast | 짧은 수정·보일러플레이트에 유리 |
| 어려운 설계 / 디버깅 | Opus 또는 GPT-5.5 | 깊은 추론이 필요한 문제 |

## 작업 유형별 추천

### 1) 기능 구현·리팩터링

일반 CRUD, 컴포넌트 추가, 타입 정리 같은 작업은 **Composer 2.5**로 충분한 경우가 많습니다. 프롬프트에 "기존 코드 스타일 유지", "변경 범위 최소화"를 넣으면 diff가 깔끔해지는 편이에요.

### 2) 아키텍처·트레이드오프 논의

"이 구조가 맞나?", "캐시를 어디에 둘까?"처럼 **판단**이 필요한 질문은 reasoning 모델이 낫습니다. Sonnet High나 Opus 계열로 후보를 2~3개 비교해 달라고 하면, 장단점 표로 정리해 주는 경우가 많아요.

### 3) 원인 불명 버그

로그·스택트레이스·재현 단계를 함께 주고, **가설 3개 + 검증 순서**를 요청하면 디버깅이 빨라집니다. 이때는 Fast 모델보다 High 계열이 유리했습니다.

### 4) 반복·대량 작업

파일 여러 개에 같은 패턴 적용, import 정리, 테스트 스켈레톤 생성 등은 **Fast** 모델이 체감상 편합니다. 품질이 살짝 떨어져도 되돌리기 쉬운 작업에 씁니다.

## 프롬프트에 넣으면 좋은 문장

```
- 변경 전에 관련 파일만 읽고, 범위를 최소화해 주세요.
- 수정 이유를 2~3문장으로 요약해 주세요.
- 테스트/빌드 명령이 있으면 실행해 보고 결과를 알려 주세요.
```

에이전트 모드에서는 위 문장이 "말만 하고 끝"인 답을 줄이는 데 도움이 됩니다.

## 비용·속도 감각

| 축 | 가벼운 모델 | 무거운 모델 |
| --- | --- | --- |
| 응답 속도 | 빠름 | 느림 |
| 추론 깊이 | 얕음 | 깊음 |
| 적합한 작업 | 반복·단순 수정 | 설계·디버깅 |
| 실수 시 비용 | 낮음 (되돌리기 쉬움) | 높음 (긴 답·과한 변경) |

매번 최고 사양을 쓰면 대화 한 턴이 길어지고, 가끔 **과한 리팩터링**이 나올 수 있어요. 그래서 저는 평소엔 가볍게, 막히면 무겁게 올리는 패턴을 씁니다.

## Hermes Agent / Cloud Agent와 함께 쓸 때

Hermes나 Cursor Cloud Agent로 자동화 파이프라인을 돌릴 때는 **모델을 고정**해 두는 게 좋습니다. 이슈 triage는 Fast, PR 본문 리뷰는 High처럼 역할별로 나누면 비용 예측이 쉬워져요.

## 피하는 습관

- 모델 이름만 바꿔 가며 **같은 질문 반복** (프롬프트를 먼저 다듬는 편이 낫습니다)
- 긴 파일 전체를 붙여 넣기 (관련 함수·에러 로그만 주는 게 정확도가 높음)
- "알아서 다 해줘"만 말하기 (완료 조건·제약을 명시할수록 결과가 안정적)

## 한 줄 정리

> 매번 최고 사양 모델 쓸 필요는 없습니다. 평소엔 가볍게, 막히면 무겁게 — 이 패턴이 제일 효율적이더라고요.

모델 이름은 Cursor 기준이며, 다른 도구에서는 이름이 다를 수 있습니다. 핵심은 **속도 vs 깊이** 트레이드오프를 의식하는 것 같아요.

비슷한 고민을 하시는 분들께 도움이 되었으면 좋겠습니다.
